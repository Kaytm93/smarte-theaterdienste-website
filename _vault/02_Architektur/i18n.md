# i18n-Strategie

## Locales
- `de` (default)
- `en`

Beide mit URL-Prefix (`/de`, `/en`) für SEO-Konsistenz und hreflang-Klarheit.

## Stack
- `next-intl` mit `defineRouting` + `pathnames`-Map
- `src/proxy.ts` = next-intl Middleware (Next.js-16-konform via `proxy`-Export)
- `getRequestConfig` lädt locale-spezifische Messages aus `src/messages/{locale}.json`
- Navigation immer via `@/lib/i18n/navigation` – nicht `next/link` direkt

## Pfad-Map (Auszug)

| Internal Key | DE | EN |
|---|---|---|
| `/projekt` | `/projekt` | `/project` |
| `/projekt/technische-standards` | `/projekt/technische-standards` | `/project/technical-standards` |
| `/beteiligung/anwendungsbeispiele` | `/beteiligung/anwendungsbeispiele` | `/participation/use-cases` |
| `/impressum` | `/impressum` | `/imprint` |

Vollständig in `src/lib/i18n/routing.ts`.

## Dynamische Inhalte
- Blog/FAQ/Termine/Partner: Übersetzungen kommen aus Supabase `*_translations`
- Query: `from('posts').select('*, post_translations!inner(*)').eq('post_translations.locale', locale)`

## SEO
- `generateMetadata` pro Layout/Page mit `getTranslations`
- hreflang-Tags via `alternates.languages` in Metadata
- `robots.txt` + `sitemap.ts` (M8)

## Switcher
- Komponente `<LanguageSwitcher>` (M2): nutzt `useRouter` + `usePathname` aus `@/lib/i18n/navigation`
- Tauscht Locale ohne Pfad-Verlust (pathnames-Map mappt automatisch)
